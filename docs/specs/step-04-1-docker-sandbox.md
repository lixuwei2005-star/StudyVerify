# StudyVerify - Step 4.1: Docker Sandbox Runner Spec

## Goal

Implement a `DockerCodeRunner` class that runs untrusted student code in a fresh
Docker container with defense-in-depth isolation. The runner adopts the existing
`SandboxRunRequest -> SandboxRunResult` contract so consumers can switch between
the subprocess runner and Docker runner without changing their public call site.

This step also refactors the existing `PythonSubprocessRunner` so test
orchestration lives in a shared abstract base class. The platform-specific
runners only own the isolation primitive:

- `PythonSubprocessRunner` executes the static wrapper in a local subprocess.
- `DockerCodeRunner` executes the static wrapper in a locked-down container.

Solver keeps using `PythonSubprocessRunner`; Verifier in Step 4.2 will use
`DockerCodeRunner`.

## Why Docker Over Subprocess

Subprocess + RLIMIT (Step 2.3) is sufficient for Solver because the code there
is LLM-generated within our prompt control. For Verifier (Step 4.2), the code
comes from students and is fundamentally untrusted. Subprocess provides no:

- Network isolation: student code could call `socket.connect()` and exfiltrate
  secrets reachable from the host process.
- Filesystem isolation: RLIMIT_NOFILE limits descriptors but does not hide the
  host filesystem.
- PID namespace isolation: subprocess fork bombs still happen in the host PID
  namespace until limits intervene.

Docker is not a magic security boundary by itself. Docker's default seccomp
profile is only moderately protective, and `user=nobody` alone is insufficient.
The security posture in this step comes from layering controls: network off,
read-only root filesystem, capped writable tmpfs, pids/cpu/memory cgroups,
ulimits, dropped Linux capabilities, `no-new-privileges`, private IPC, non-root
user, and short wall-clock timeouts.

## Existing Public Contract

The current repo already defines the public sandbox request/result types:

```python
# app/sandbox/schemas.py

class SandboxRunRequest(BaseModel):
    code: str
    entry_function: str
    test_cases: list[dict]
    timeout_seconds: int = 5
    memory_mb: int = 128


class SandboxRunResult(BaseModel):
    status: Literal["all_passed", "some_failed", "error", "timeout"]
    test_results: list[TestExecutionResult] = Field(default_factory=list)
    pass_count: int = 0
    fail_count: int = 0
    error: str | None = None
```

`SandboxRunResult` intentionally does not expose raw `stdout`, `stderr`,
`exit_code`, or `timed_out`. Those are runner-internal execution details used
to build the existing result shape.

The existing public runner method is:

```python
class PythonSubprocessRunner:
    async def run(self, request: SandboxRunRequest) -> SandboxRunResult:
        ...
```

`DockerCodeRunner` must use the same public method shape:

```python
async def run(self, request: SandboxRunRequest) -> SandboxRunResult
```

There is no `stdin_data`, `payload`, `timeout`, or `memory_limit` parameter on
the public API.

## Abstract Base Class: `TestRunner`

Create a new shared base class in `backend/app/sandbox/base_runner.py`.

```python
# app/sandbox/base_runner.py

from abc import ABC, abstractmethod

from app.sandbox.schemas import SandboxRunRequest, SandboxRunResult


class TestRunner(ABC):
    """Abstract base for sandbox runners.

    Subclasses implement _execute_code (the platform-specific isolation
    primitive); the base class owns the test orchestration loop:
    iterate test_cases, dispatch each test, compare actual to expected,
    and aggregate to SandboxRunResult.
    """

    async def run(self, request: SandboxRunRequest) -> SandboxRunResult:
        # Common test orchestration:
        # - For each test_case in request.test_cases:
        #   - Build payload (code + entry_function + this test's input)
        #   - Call self._execute_code(wrapper_code, payload, timeout, memory)
        #   - Parse stdout JSON, compare actual to expected
        #   - Append TestExecutionResult
        # - Aggregate status (all_passed / some_failed / error / timeout)
        # - Return SandboxRunResult
        ...

    @abstractmethod
    async def _execute_code(
        self,
        code: str,
        payload: bytes,
        timeout_seconds: int,
        memory_mb: int,
    ) -> tuple[bytes, bytes, int, bool]:
        """Run code with payload as stdin.

        Returns:
            (stdout, stderr, exit_code, timed_out)
        """
        ...
```

### Base-runner responsibilities

`TestRunner.run()` owns all runner-agnostic logic:

- Build a per-test JSON payload with:
  - `code`: student source from `request.code`
  - `entry_function`: function name to call
  - `input`: one test case's input string
- Execute a static wrapper script through `_execute_code(...)`.
- Parse wrapper stdout as JSON.
- Compare the returned `actual` against the test case's `expected`, using the
  existing `repr(actual) == expected or str(actual) == expected` behavior.
- Produce `TestExecutionResult` rows.
- Aggregate:
  - `timeout` if any execution reports `timed_out=True`
  - `error` for fatal wrapper/runner errors
  - `all_passed` if all test cases pass
  - `some_failed` otherwise

The static wrapper remains trusted code. Student code is data inside the stdin
payload, then `exec(...)`ed by the wrapper inside the chosen isolation boundary.

### Refactor `PythonSubprocessRunner`

Keep the existing class name for backward compatibility:

```python
# app/sandbox/runner.py

class PythonSubprocessRunner(TestRunner):
    async def _execute_code(
        self,
        code: str,
        payload: bytes,
        timeout_seconds: int,
        memory_mb: int,
    ) -> tuple[bytes, bytes, int, bool]:
        ...
```

Move the current loop over `request.test_cases`, stdout parsing, pass/fail
comparison, and `SandboxRunResult` aggregation from `runner.py` into
`base_runner.py`. Leave only the subprocess-specific `subprocess.run(...)`
call, tempdir, timeout handling, and rlimit setup in
`PythonSubprocessRunner._execute_code(...)`.

Backward compatibility: Solver code that calls `runner.run(request)` continues
to work unchanged because the public signature is preserved.

## DockerCodeRunner Implementation

```python
# app/sandbox/docker_runner.py

from __future__ import annotations

import asyncio
import logging
import socket as stdlib_socket
import tempfile
import time
from pathlib import Path

import docker
import requests
from docker.errors import APIError, ContainerError, ImageNotFound, NotFound

from app.sandbox.base_runner import TestRunner

logger = logging.getLogger(__name__)


class DockerCodeRunner(TestRunner):
    """Run the static test wrapper in an isolated Docker container.

    Each _execute_code() call creates and tears down a fresh container. No state
    persists between individual test executions.

    Container hardening, expressed as Docker CLI equivalents:
    - --network=none                : No network access
    - --read-only                   : Root filesystem read-only
    - --tmpfs /tmp:size=64m,noexec  : Writable scratch, capped, no exec
    - --memory=<N>m                 : Hard memory cap (OOM kills)
    - --memory-swap=<N>m            : Disables swap (no overflow)
    - --pids-limit=32               : Cap on processes (anti-fork-bomb)
    - --cpus=1.0                    : Single CPU core
    - --shm-size=8m                 : Limit /dev/shm
    - --cap-drop=ALL                : Drop all Linux capabilities
    - --security-opt=no-new-privileges:true : Block setuid escalation
    - --ipc=private                 : Private IPC namespace
    - --user=nobody                 : Non-root user
    - --init                        : Use tini PID 1 to reap zombies
    - --label studyverify=sandbox   : Tag for cleanup queries
    - --rm                          : Implemented by explicit remove() cleanup
    - ulimits=[                     : Belt-and-suspenders
        {nproc:32, hard:32},          in addition to pids-limit
        {nofile:64, hard:64},
      ]

    The mounted /sandbox/code.py is trusted wrapper code. Student code and the
    current test case flow through stdin JSON.
    """

    DEFAULT_IMAGE = "python:3.11-slim"

    def __init__(
        self,
        image: str = DEFAULT_IMAGE,
        client: docker.DockerClient | None = None,
    ) -> None:
        self.image = image
        self._client = client or docker.from_env()

    async def _execute_code(
        self,
        code: str,
        payload: bytes,
        timeout_seconds: int,
        memory_mb: int,
    ) -> tuple[bytes, bytes, int, bool]:
        return await asyncio.to_thread(
            self._execute_code_sync,
            code,
            payload,
            timeout_seconds,
            memory_mb,
        )
```

`DockerCodeRunner` inherits `run(request: SandboxRunRequest) ->
SandboxRunResult` from `TestRunner`. It does not expose a separate public
`run(code, stdin_data, ...)` method.

## Implementation Notes

### 1. Write trusted wrapper code to a temp file

```python
with tempfile.NamedTemporaryFile(mode="w", suffix=".py", delete=False) as f:
    f.write(code)
    host_code_path = Path(f.name)
```

On macOS, `tempfile` typically uses `/var/folders/...`, which Docker Desktop
usually has filesystem-sharing access to. If a developer changes Docker
Desktop file-sharing settings, integration tests may fail with a bind-mount
error; that is an environment issue, not a sandbox implementation issue.

### 2. Create container with full hardening

Use `containers.create(...)`, not `containers.run(...)`, so stdin can be
attached before the container starts.

```python
container = self._client.containers.create(
    image=self.image,
    command=["python", "/sandbox/code.py"],
    stdin_open=True,
    detach=True,
    network_mode="none",
    read_only=True,
    tmpfs={"/tmp": "size=64m,noexec"},
    mem_limit=f"{memory_mb}m",
    memswap_limit=f"{memory_mb}m",
    pids_limit=32,
    nano_cpus=1_000_000_000,
    shm_size="8m",
    cap_drop=["ALL"],
    security_opt=["no-new-privileges:true"],
    ipc_mode="private",
    user="nobody",
    init=True,
    labels={"studyverify": "sandbox"},
    ulimits=[
        docker.types.Ulimit(name="nproc", soft=32, hard=32),
        docker.types.Ulimit(name="nofile", soft=64, hard=64),
    ],
    volumes={
        str(host_code_path): {
            "bind": "/sandbox/code.py",
            "mode": "ro",
        }
    },
)
```

The CLI-equivalent `--rm` behavior is implemented by the `finally` block's
explicit `container.remove(force=True)`. Do not use `auto_remove=True` here:
the container might disappear before logs are collected.

### 3. Attach stdin before start

The important ordering is create -> attach -> start. Starting first and then
attaching can race if the container exits quickly.

```python
container = None
sock = None
try:
    # 1. create() does NOT start the container
    container = self._client.containers.create(...)

    # 2. attach socket BEFORE start so stdin is ready
    sock = container.attach_socket(
        params={"stdin": 1, "stream": 1}
    )

    # 3. NOW start the container
    container.start()

    # 4. write stdin payload, close write side
    sock._sock.sendall(payload)  # type: ignore[attr-defined]
    sock._sock.shutdown(stdlib_socket.SHUT_WR)

    # 5. wait for exit with timeout
    try:
        result = container.wait(timeout=timeout_seconds)
        exit_code = result["StatusCode"]
        timed_out = False
    except (requests.exceptions.ReadTimeout, ConnectionError):
        container.kill()
        # Brief settle delay before reading logs to avoid racing with kill
        # cleanup and log flushing.
        time.sleep(0.2)
        try:
            container.wait(timeout=2)
        except Exception:
            pass
        exit_code = -1
        timed_out = True

    # 6. fetch logs after the container is dead
    stdout = container.logs(stdout=True, stderr=False)
    stderr = container.logs(stdout=False, stderr=True)

finally:
    # 7. cleanup, defensive - handle every failure mode
    if sock is not None:
        try:
            sock.close()
        except Exception:
            pass
    if container is not None:
        try:
            container.remove(force=True)
        except (APIError, NotFound):
            pass  # already gone
    host_code_path.unlink(missing_ok=True)
```

`sock._sock` is a private SDK attribute, but the Docker SDK does not provide a
clean public `input=` API for detached-container stdin. Keep the
`# type: ignore[attr-defined]` comment at the access site.

### 4. Return raw execution details to the base class

```python
return stdout, stderr, exit_code, timed_out
```

`TestRunner.run()` converts those raw details into `SandboxRunResult`.

## Cancellation Semantics

Async cancellation does not kill the running container. When the caller cancels
the asyncio task wrapping `run()`, the underlying thread created by
`asyncio.to_thread(...)` continues executing until `container.wait()` times out
and the `finally` block runs cleanup. This means:

- If a Verifier session is canceled, the student's container keeps running up
  to `timeout_seconds`.
- The container will be cleaned up correctly via the `finally` block; no
  orphans.
- There is no way to truly abort a `docker.containers.wait()` call in flight.

This is acceptable for our use case because timeouts are bounded at 5-10s by
default, but consumers of `DockerCodeRunner` should not assume
`cancel() == immediate stop`.

## Test Strategy

### Unit tests: `test_docker_runner_unit.py`

Mock `docker.DockerClient` or the specific methods used:

- `test_runner_passes_correct_isolation_flags` - assert
  `containers.create(...)` receives `network_mode="none"`, `read_only=True`,
  `cap_drop=["ALL"]`, `security_opt=["no-new-privileges:true"]`, `pids_limit`,
  memory limits, `ulimits`, `init=True`, and the label.
- `test_runner_uses_create_attach_start_order` - assert the order is
  `create()`, `attach_socket()`, `start()`, write stdin, wait.
- `test_runner_pipes_stdin_payload` - mock the attached socket and verify exact
  JSON bytes are sent.
- `test_runner_returns_sandbox_run_result_on_success` - mock stdout/stderr/exit
  code and assert `SandboxRunResult` is populated through the base class.
- `test_runner_handles_timeout` - mock `container.wait` raising
  `requests.exceptions.ReadTimeout`, assert `timed_out=True`,
  `container.kill()` is called, and cleanup still runs.
- `test_runner_cleans_up_on_create_exception` - mock `containers.create`
  raising `APIError`; assert temp file cleanup and no container removal attempt
  against an uninitialized object.
- `test_runner_cleans_up_on_attach_or_start_exception` - assert socket and
  container cleanup are defensive.
- `test_runner_uses_custom_image` - pass `image="my-image:tag"` and verify it
  propagates.

### Integration tests: `test_docker_runner_integration.py`

Require a real Docker daemon. Skip if `docker.from_env().ping()` fails.

- `test_happy_path_round_trip` - simple function, one or more test cases,
  result is `all_passed`.
- `test_timeout_kills_long_running_code` - `while True: pass`; assert
  `status == "timeout"` and wall time is bounded.
- `test_oom_caught_by_memory_limit` - allocate far beyond `memory_mb`; assert
  non-success and not an unbounded hang.
- `test_baseline_network_isolation` - code attempts
  `socket.create_connection(("8.8.8.8", 53))`; assert failure.
- `test_baseline_filesystem_readonly` - code attempts `open("/etc/test", "w")`;
  assert failure.
- `test_tmpfs_writable` - code writes and reads `/tmp/scratch`; assert success.
- `test_concurrent_runs_isolated` - run 3+ requests via `asyncio.gather`; each
  sees only its own `/tmp` state.
- `test_no_orphan_containers_after_run` - count containers by label before and
  after; count must not increase.
- `test_dns_resolution_blocked` - code attempts DNS resolution such as
  `socket.getaddrinfo("example.com", 80)`; assert failure under
  `network_mode="none"`.
- `test_fork_bomb_caught_by_pids` - fork/spawn loop hits `pids_limit=32` or
  `nproc` ulimit; assert bounded failure.
- `test_stdout_flood_truncated` - student prints a large amount of output
  (for example 100 MB); verify the app process does not OOM or fill disk and
  stored error text is truncated.
- `test_huge_stdin_handled` - 1 MB stdin payload is accepted and processed.
- `test_capability_drop_blocks_mount` - student tries
  `os.system("mount -t tmpfs tmpfs /tmp/m")`; verify failure/EPERM due to
  `cap_drop=["ALL"]` and `no-new-privileges`.

These are baseline isolation smoke tests, not a formal proof of container
escape resistance.

## Apple Silicon / Docker Desktop Notes

- `python:3.11-slim` is multi-arch, so it should run natively on arm64 Docker
  Desktop without forcing `platform=linux/amd64`.
- Avoid hard-coding architecture in Step 4.1 tests unless a test explicitly
  validates platform selection.
- Docker resource limits run inside Docker Desktop's Linux VM on macOS. They
  should still be asserted through observed behavior, but failures may reflect
  Docker Desktop settings rather than host macOS kernel behavior.
- Bind mounts from `tempfile.NamedTemporaryFile` paths under `/var/folders/...`
  are normally shared by Docker Desktop. If not, configure Docker Desktop file
  sharing or move the temp directory under a shared path for local tests.

## What NOT to Do

- DO NOT use `subprocess.run("docker run ...")`; use the Docker SDK.
- DO NOT add a new public `run(code, stdin_data, ...)` API; keep
  `run(request: SandboxRunRequest) -> SandboxRunResult`.
- DO NOT introduce a separate raw stdout/stderr result type; the repo's
  existing result type is `SandboxRunResult`.
- DO NOT rename `PythonSubprocessRunner`; Solver imports should continue to
  work.
- DO NOT skip `network_mode="none"`.
- DO NOT skip `cap_drop=["ALL"]`; `user=nobody` alone is not sufficient, and
  default capabilities still allow some capability-sensitive operations.
- DO NOT skip `security_opt=["no-new-privileges:true"]`; without it, setuid
  binaries inside the image could let student code escalate.
- DO NOT use `containers.run(detach=True)` followed by attach; that races.
  Always use `create() -> attach_socket() -> start()`.
- DO NOT call `container.logs()` immediately after `kill()`; wait briefly for
  kernel/container cleanup to settle and logs to flush.
- DO NOT initialize `container` as anything other than `None` at the top of the
  try block; if `create()` fails, the `finally` block needs a sentinel.
- DO NOT use `auto_remove=True`; logs must be collected before explicit
  removal.
- DO NOT use `--privileged` or mount host directories beyond the single
  read-only wrapper file.
- DO NOT mount `docker.sock` yet; that is a Step 4.4 compose change.
- DO NOT pull the image at runtime; pre-pull as an ops/test setup step and
  raise `ImageNotFound` clearly if absent.
- DO NOT reuse containers across runs; a fresh container per execution is the
  isolation invariant.
