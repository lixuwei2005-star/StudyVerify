# StudyVerify — Step 2.3: Sandbox Self-Verification Spec

## Goal
Add a self-verification stage to `SolverAgent` so that the generated 
code is **executed against the provided test cases** before being 
returned. Failed runs trigger one retry of the Code stage; a second 
failure returns a degraded result (low confidence, `verified=false`) 
rather than raising.

This closes the Solver's "self-truth" loop: the Solver never returns 
unverified code claiming high confidence.

## Why This Matters (architectural rationale)
- Solver output is the ground truth for Verifier (Step 4) and Hint 
  (Step 5). Unverified Solver output poisons the entire pipeline.
- LLMs occasionally produce syntactically valid but semantically 
  wrong code. Running the code is the only way to catch this.
- This is the project's core thesis ("don't trust the LLM as judge") 
  applied to the Solver itself.

## Scope: Subprocess-Based Sandbox (NOT Docker)

This step uses Python `subprocess` with timeout + isolated tempdir. 
**Docker-based sandbox is deferred to Step 4** when we add the 
Verifier Agent (which runs UNTRUSTED student code, the real attack 
surface). Solver code is generated by our own LLM call, lower-risk.

Document this trade-off in code comments + commit message.

## Tech Stack Additions
- `subprocess` (stdlib) — process isolation
- `tempfile` (stdlib) — per-run scratch directory
- `resource` (stdlib, POSIX-only) — RLIMIT_AS for memory cap on macOS/Linux
- No new pip dependencies

## Schemas

### `app/sandbox/schemas.py`

````python
from pydantic import BaseModel, Field
from typing import Literal

class SandboxRunRequest(BaseModel):
    code: str  # full Python code, must define expected entry function
    entry_function: str  # e.g., "sum_list"
    test_cases: list[dict]  # [{"input": "[1,2,3]", "expected": "6"}, ...]
    timeout_seconds: int = 5
    memory_mb: int = 128

class TestExecutionResult(BaseModel):
    test_index: int
    input: str
    expected: str
    actual: str | None = None  # None if execution errored
    passed: bool
    error: str | None = None  # exception type + message if errored
    duration_ms: int

class SandboxRunResult(BaseModel):
    status: Literal["all_passed", "some_failed", "error", "timeout"]
    test_results: list[TestExecutionResult]
    pass_count: int
    fail_count: int
    error: str | None = None  # only set when status=="error" or "timeout"
````

## SubprocessRunner Implementation Requirements

`PythonSubprocessRunner.run(request: SandboxRunRequest) -> SandboxRunResult`:

### Process isolation
- Use `subprocess.run()` with explicit `stdin`, `stdout`, `stderr` 
  pipes (text mode UTF-8)
- Spawn `python -c <wrapper_script>` — DO NOT write user code to a 
  file and execute by path (slower, leaves traces)
- Use `cwd=` pointing to a fresh `tempfile.mkdtemp()` to isolate 
  filesystem; clean up in `finally`

### Resource limits (POSIX)
- Use `preexec_fn=` to set:
  - `resource.setrlimit(RLIMIT_AS, (memory_mb*1024*1024, ...))` — 
    address space cap
  - `resource.setrlimit(RLIMIT_CPU, (timeout_seconds + 1, ...))` — 
    CPU time cap (defense-in-depth, primary timeout still on subprocess)
- On Windows, log warning and skip rlimit (resource module unavailable). 
  Document this limitation; project is POSIX-first.

### Timeout
- `subprocess.run(timeout=request.timeout_seconds)`. On 
  `subprocess.TimeoutExpired`:
  - Set `status="timeout"`, return result with no test_results 
    (since we couldn't run any)

### Wrapper script (固定模板,数据走 stdin — 不可注入)

The wrapper is a STATIC Python script. It does NOT contain any 
templated code. The runner passes a JSON payload via stdin 
containing the user code, entry function name, and test cases.

The runner spawns the subprocess as:
```
[python, "-c", WRAPPER_SCRIPT]
```
with `input=json.dumps(payload).encode("utf-8")` written to stdin 
in `subprocess.run(..., input=..., capture_output=True)`.

The wrapper:
1. Reads stdin, parses as JSON: `{"code": str, "entry_function": str, "test_cases": list}`
2. `exec()`s the user code into a clean namespace dict
3. Looks up the entry function in that namespace
4. For each test case: parses input via `ast.literal_eval` (NOT `eval`), 
   calls the function, compares result to expected
5. Outputs JSON list of test results to stdout
6. Wraps each test in try/except — exception in one test does NOT 
   stop the others
7. Function-not-defined or stdin-parse-error → exit code 2 with 
   diagnostic on stderr; runner maps to `status="error"`

Reference wrapper (Claude Code may refine):

```python
WRAPPER_SCRIPT = r'''
import ast
import json
import sys
import time

def _main():
    try:
        payload = json.loads(sys.stdin.read())
        user_code = payload["code"]
        entry_function = payload["entry_function"]
        test_cases = payload["test_cases"]
    except Exception as e:
        print(f"FATAL: bad payload: {type(e).__name__}: {e}", file=sys.stderr)
        sys.exit(2)

    namespace = {}
    try:
        exec(user_code, namespace)
    except Exception as e:
        print(f"FATAL: user code failed to load: {type(e).__name__}: {e}", file=sys.stderr)
        sys.exit(2)

    fn = namespace.get(entry_function)
    if not callable(fn):
        print(f"FATAL: function {entry_function} not defined or not callable", file=sys.stderr)
        sys.exit(2)

    results = []
    for i, tc in enumerate(test_cases):
        start = time.perf_counter()
        try:
            args = ast.literal_eval(tc["input"])
            actual = fn(*args) if isinstance(args, tuple) else fn(args)
            actual_str = repr(actual)
            passed = actual_str == tc["expected"] or str(actual) == tc["expected"]
            results.append({
                "test_index": i,
                "input": tc["input"],
                "expected": tc["expected"],
                "actual": actual_str,
                "passed": passed,
                "error": None,
                "duration_ms": int((time.perf_counter() - start) * 1000),
            })
        except Exception as e:
            results.append({
                "test_index": i,
                "input": tc["input"],
                "expected": tc["expected"],
                "actual": None,
                "passed": False,
                "error": f"{type(e).__name__}: {e}",
                "duration_ms": int((time.perf_counter() - start) * 1000),
            })

    print(json.dumps(results))

_main()
'''
```

The wrapper above is a CONSTANT in `app/sandbox/runner.py` and never 
changes per request. User code, entry function, and test cases flow 
through stdin as JSON only.

### Subprocess invocation

```python
proc = subprocess.run(
    [sys.executable, "-c", WRAPPER_SCRIPT],
    input=json.dumps({
        "code": request.code,
        "entry_function": request.entry_function,
        "test_cases": request.test_cases,
    }),
    capture_output=True,
    text=True,
    timeout=request.timeout_seconds,
    cwd=tempdir,
    preexec_fn=_set_rlimits(memory_mb, timeout_seconds),  # POSIX only
)
```

Parsing rules:
- `proc.returncode == 0` and stdout parses as JSON list → success
- `proc.returncode == 2` (wrapper FATAL) → `status="error"`, `error` 
  field = stderr content (truncated to 500 chars)
- Other non-zero return code → `status="error"`, generic message + 
  stderr captured
- `subprocess.TimeoutExpired` → `status="timeout"`
- JSON parse fails on stdout → `status="error"`, include first 500 
  chars of stdout in error field

### Parsing
- Read stdout as JSON. If parse fails (subprocess crashed before 
  emitting), return `status="error"` with stderr captured.
- If wrapper exited with non-zero code without JSON, return `status="error"`.

### Logging
- Per-run structured log: `sandbox.run` with: 
  `entry_function`, `n_tests`, `pass_count`, `fail_count`, `status`, 
  `wall_ms`, `subprocess_returncode`.

## Solver Agent Updates

### New flow in `SolverAgent.solve()`:
1. Existing: analyze → plan → code → produce candidate `SolverOutput`
2. **Extract `entry_function` name from `code`** (regex `def (\w+)\(`, 
   first match — Solver prompt already enforces single function)
3. Run `PythonSubprocessRunner.run(...)`
4. Branch on result:
   - `all_passed` → set `verified=true`, attach `test_results`, return
   - `some_failed` or `error` (NOT `timeout`) → if first attempt, 
     regenerate code via `build_code_retry_messages(...)` with error 
     details in prompt, run sandbox again
   - `timeout` → no retry (likely infinite loop, regenerating won't 
     help quickly), set `verified=false`, attach empty test_results, 
     return with degraded confidence
5. After retry: if still failing, set `verified=false`, attach final 
   `test_results`, return.

### Retry prompt (`build_code_retry_messages`)
Same as `build_code_messages` but appends to user message:

````
Your previous attempt failed sandbox verification:

Code:
```python
{previous_code}
```

Test results:
{formatted_test_failures}

Generate a corrected version. Return JSON object with keys 'code' 
and 'explanation'. Address the specific failures above.
````

## Test Requirements

### `tests/sandbox/test_runner.py` — pure sandbox tests, NO LLM
Cover at minimum:
- Happy path: simple `def add(a, b): return a + b` with 3 test cases, 
  all pass
- Function not defined → `status="error"`
- Test case raises exception → recorded in `test_results[i].error`, 
  other tests still run

  wall time
- Memory bomb (`[0] * 10**9`) → MemoryError captured OR process 
  killed (either acceptable; assert NOT all_passed)
- Wrong answer → `passed=false`, `actual` populated
- Multi-arg function via tuple input: `"(2, 3)"` → `fn(2, 3)`

## What NOT to do
- DO NOT use `eval()` for parsing test inputs — use `ast.literal_eval`
- DO NOT use Docker — that's Step 4
- DO NOT write user code to disk — pass via wrapper template + stdin/argv
- DO NOT skip resource limits "because it's just our own LLM code"
- DO NOT retry on timeout (waste of API + time, almost certainly 
  infinite loop)
- DO NOT silence sandbox failures and pretend `verified=true`
- DO NOT add network restrictions yet (subprocess can't easily; defer 
  to Docker step)

- DO NOT inject user code into wrapper source via f-string / .format() 
  / template substitution. Wrapper is constant; user code flows 
  through stdin as JSON data.
