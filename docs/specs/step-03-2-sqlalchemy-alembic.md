# StudyVerify — Step 3.2: SQLAlchemy 2.0 async + Alembic Spec

## Goal
Wire the FastAPI backend to the Postgres container from Step 3.1 
using SQLAlchemy 2.0 async, manage schema with Alembic, and verify 
end-to-end CRUD against both real Postgres (integration) and 
in-memory SQLite (unit tests).

This step does NOT touch SolverAgent. It only builds the data layer 
that 3.3 will plug into.

## Why This Step Is Narrow
SQLAlchemy + Alembic + async + dependency injection is a lot of 
moving parts. Isolating "just the data layer works" before changing 
agent code keeps debugging surface small.

## SQLAlchemy 2.0 Patterns to Follow

### Naming convention (in `db/base.py`)
This is critical for Alembic to generate stable, descriptive 
constraint names — without it, autogen produces random names that 
make migrations noisy.

```python
from sqlalchemy import MetaData
from sqlalchemy.orm import DeclarativeBase

NAMING_CONVENTION = {
    "ix": "ix_%(column_0_label)s",
    "uq": "uq_%(table_name)s_%(column_0_name)s",
    "ck": "ck_%(table_name)s_%(constraint_name)s",
    "fk": "fk_%(table_name)s_%(column_0_name)s_%(referred_table_name)s",
    "pk": "pk_%(table_name)s",
}

class Base(DeclarativeBase):
    metadata = MetaData(naming_convention=NAMING_CONVENTION)
```

### Async engine + session factory (`db/session.py`)
```python
from sqlalchemy.ext.asyncio import (
    create_async_engine, async_sessionmaker, AsyncSession
)
from app.core.config import get_settings

_engine = None
_session_factory = None

def get_engine():
    global _engine
    if _engine is None:
        settings = get_settings()
        if not settings.DATABASE_URL:
            raise RuntimeError(
                "DATABASE_URL is required. Set it in backend/.env, e.g. "
                "postgresql+asyncpg://studyverify:<password>@localhost:5432/studyverify"
            )
        _engine = create_async_engine(
            settings.DATABASE_URL,
            echo=settings.DB_ECHO_SQL,
            pool_size=settings.DB_POOL_SIZE,
            max_overflow=settings.DB_MAX_OVERFLOW,
            pool_timeout=settings.DB_POOL_TIMEOUT_SECONDS,
            pool_pre_ping=True,  # detects stale connections after restart
        )
    return _engine

def get_session_factory():
    global _session_factory
    if _session_factory is None:
        _session_factory = async_sessionmaker(
            get_engine(),
            class_=AsyncSession,
            expire_on_commit=False,  # critical for FastAPI: lets you 
                                     # use objects after session closes
        )
    return _session_factory

async def get_db_session():
    """FastAPI dependency. Yields an AsyncSession.
    
    Does NOT auto-commit. Routes/services must call session.commit() 
    explicitly when they want changes persisted. This makes transaction 
    boundaries explicit in business code, avoiding hidden writes from 
    GET requests or partial-success scenarios.
    
    On exception, the session is rolled back automatically.
    """
    async with get_session_factory()() as session:
        try:
            yield session
        except Exception:
            await session.rollback()
            raise

async def dispose_engine():
    """Close the engine and reset globals. Called on app shutdown 
    and in test fixtures."""
    global _engine, _session_factory
    if _engine is not None:
        await _engine.dispose()
        _engine = None
        _session_factory = None
```

Key points:
- `pool_pre_ping=True` — survives Postgres container restarts
- `expire_on_commit=False` — required by FastAPI usage pattern; 
  without this, accessing object attrs after commit raises 
  `MissingGreenlet` errors
- Lazy globals — engine/factory created on first use, not at import 
  (lets test fixtures override)
- Context-manager dependency — auto rollback on exception
- Explicit transaction boundaries — routes/services call 
  `session.commit()` only when they want changes persisted
- Engine disposal hook — FastAPI lifespan and tests can close pooled 
  connections cleanly

### Model declaration (`db/models.py`)
```python
import uuid
from datetime import datetime
from decimal import Decimal
from sqlalchemy import String, Text, Boolean, Numeric, DateTime, Uuid, func
from sqlalchemy.types import JSON
from sqlalchemy.dialects.postgresql import JSONB
from sqlalchemy.orm import Mapped, mapped_column

from app.db.base import Base

JSONType = JSONB().with_variant(JSON(), "sqlite")


class SolverSession(Base):
    __tablename__ = "solver_sessions"

    id: Mapped[uuid.UUID] = mapped_column(
        Uuid(as_uuid=True, native_uuid=True),
        primary_key=True,
        default=uuid.uuid4,
    )
    problem_id: Mapped[str] = mapped_column(String(64), nullable=False, index=True)
    problem_text: Mapped[str] = mapped_column(Text, nullable=False)
    test_cases: Mapped[list] = mapped_column(JSONType, nullable=False)
    
    analysis: Mapped[str] = mapped_column(Text, nullable=False)
    plan_steps: Mapped[list] = mapped_column(JSONType, nullable=False)
    code: Mapped[str] = mapped_column(Text, nullable=False)
    explanation: Mapped[str] = mapped_column(Text, nullable=False)
    
    verified: Mapped[bool] = mapped_column(Boolean, nullable=False, index=True)
    test_results: Mapped[list] = mapped_column(JSONType, nullable=False)
    confidence: Mapped[Decimal] = mapped_column(Numeric(3, 2), nullable=False)
    
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), server_default=func.now(), nullable=False, index=True
    )

    def __repr__(self) -> str:
        return f"<SolverSession {self.id} {self.problem_id} verified={self.verified}>"
```

⚠️ **Cross-database compat note**: `JSONB` is Postgres-specific. 
SQLite tests will fail unless we handle this. Solution: use a type 
variant:

```python
from sqlalchemy.types import JSON
from sqlalchemy.dialects.postgresql import JSONB

# JSONB on Postgres, JSON on SQLite — same Python interface
JSONType = JSONB().with_variant(JSON(), "sqlite")
```

Then use `JSONType` in the model. For UUIDs, use SQLAlchemy 2.0's
`sqlalchemy.Uuid(as_uuid=True, native_uuid=True)` directly; it handles
dialect differences without a SQLite variant hack.

## Alembic Configuration

### `alembic.ini` (key settings)
- `script_location = alembic`
- `sqlalchemy.url` is OVERRIDDEN at runtime in `env.py` — the value 
  in alembic.ini is just a placeholder. This lets DATABASE_URL flow 
  from Settings.

### `alembic/env.py` (replace default with async-aware version)

The default `alembic init async ...` template is close but needs 
customization:
- Import `Base` from `app.db.base`
- Set `target_metadata = Base.metadata`  
- Read `DATABASE_URL` from Settings, NOT alembic.ini
- Use `create_async_engine` for online mode
- Use `compare_type=True` and `compare_server_default=True` for 
  better autogen detection

Reference structure:
```python
import asyncio
from logging.config import fileConfig
from sqlalchemy.ext.asyncio import async_engine_from_config
from sqlalchemy import pool
from alembic import context

from app.core.config import get_settings
from app.db.base import Base
from app.db import models  # ← ensures models are registered

config = context.config
settings = get_settings()
config.set_main_option("sqlalchemy.url", settings.DATABASE_URL)

if config.config_file_name is not None:
    fileConfig(config.config_file_name)

target_metadata = Base.metadata

def do_run_migrations(connection):
    context.configure(
        connection=connection,
        target_metadata=target_metadata,
        compare_type=True,
        compare_server_default=True,
    )
    with context.begin_transaction():
        context.run_migrations()

async def run_async_migrations():
    connectable = async_engine_from_config(
        config.get_section(config.config_ini_section, {}),
        prefix="sqlalchemy.",
        poolclass=pool.NullPool,
    )
    async with connectable.connect() as connection:
        await connection.run_sync(do_run_migrations)
    await connectable.dispose()

def run_migrations_online():
    asyncio.run(run_async_migrations())

if context.is_offline_mode():
    raise NotImplementedError("Offline mode not supported")
else:
    run_migrations_online()
```

## FastAPI Dependency Pattern

### `app/api/routes/health_db.py`
```python
from fastapi import APIRouter, Depends
from sqlalchemy import text
from sqlalchemy.ext.asyncio import AsyncSession

from app.db.session import get_db_session

router = APIRouter(prefix="/health", tags=["health"])

@router.get("/db")
async def health_db(session: AsyncSession = Depends(get_db_session)):
    result = await session.execute(text("SELECT 1"))
    value = result.scalar_one()
    return {"status": "ok" if value == 1 else "error", "db": "reachable"}
```

This is the pattern that 3.3 will use for solver persistence — 
inject session via `Depends(get_db_session)`.

Test cases (must cover):
1. Create + read SolverSession — round-trip preserves all fields 
   including JSONB lists and UUID
2. Filter by `problem_id` — index works, returns expected row
3. Filter by `verified=False` — returns only unverified rows
4. Update `verified` — change persists

### Integration tests (`tests/db/test_models_integration.py`)
- `@pytest.mark.integration`
- Skips if `DATABASE_URL` not set in env
- Connects to real Postgres (Step 3.1 container must be up)
- Test: create + read SolverSession against real DB
- Test: JSONB storage works (list of dicts round-trips)
- Test: UUID is generated as native UUID, not CHAR

⚠️ Integration test must NOT pollute prod DB. Use a 
`pytest_asyncio.fixture` that creates a temporary schema per test and 
drops it afterward.

Schema-isolation chosen over savepoint pattern for fixture simplicity. 
Each integration test creates a fresh `test_<uuid>` schema, runs 
against it, then drops the schema. Trade-off: ~50ms per test instead 
of <5ms, but fixture is half the code and debug-friendly. Revisit if 
integration suite grows beyond ~30 tests.

Pseudocode:
```python
import uuid

import pytest
import pytest_asyncio
from sqlalchemy import text
from sqlalchemy.ext.asyncio import AsyncSession, create_async_engine

from app.core.config import get_settings
from app.db import models as _models  # noqa: F401; register metadata
from app.db.base import Base


@pytest_asyncio.fixture
async def integration_session():
    settings = get_settings()
    if not settings.DATABASE_URL:
        pytest.skip("DATABASE_URL not set")

    schema_name = f"test_{uuid.uuid4().hex}"
    engine = create_async_engine(settings.DATABASE_URL)

    async with engine.connect() as conn:
        await conn.execute(text(f'CREATE SCHEMA "{schema_name}"'))
        await conn.execute(text(f'SET search_path TO "{schema_name}"'))
        await conn.run_sync(Base.metadata.create_all)
        await conn.commit()

        try:
            await conn.execute(text(f'SET search_path TO "{schema_name}"'))
            async with AsyncSession(bind=conn, expire_on_commit=False) as session:
                yield session
        finally:
            await conn.rollback()
            await conn.execute(text(f'DROP SCHEMA IF EXISTS "{schema_name}" CASCADE'))
            await conn.commit()

    await engine.dispose()
```

## What NOT to do
- DO NOT use sync SQLAlchemy patterns (`Session()`, `engine.connect()`) 
  — async only
- DO NOT hardcode DATABASE_URL anywhere — env-driven only
- DO NOT skip `expire_on_commit=False` — FastAPI usage WILL break 
  without it
- DO NOT use `Base.metadata.create_all()` in app code — Alembic 
  owns schema. (OK in test fixtures only.)
- DO NOT commit `alembic.ini` with a real DB URL — placeholder only
- DO NOT use `JSONB` directly without the `JSONType` variant 
  pattern — kills SQLite tests
- DO NOT use `sqlalchemy.dialects.postgresql.UUID` with `.with_variant()` 
  for SQLite compat. Use SQLAlchemy 2.0's `sqlalchemy.Uuid` type 
  with `native_uuid=True` instead — it auto-handles dialect differences.
- DO NOT auto-commit in the `get_db_session` dependency. Sessions 
  yield, then close. Commits are explicit in business code.
- DO NOT default `DATABASE_URL` to a fake value like 
  `"postgresql+asyncpg://user:CHANGEME@..."`. Empty string + loud-fail 
  is clearer than "auth failed" errors 30 minutes later.
- DO NOT skip the FastAPI lifespan / `engine.dispose()` — long-running 
  servers leak connections without it.
- DO NOT write raw SQL in business code — use Core `select()` or 
  ORM. (Raw SQL in health check via `text()` is the one exception.)
- DO NOT mark a migration "tested" without trying both 
  upgrade AND downgrade
- DO NOT skip the alembic naming_convention setup — autogen 
  quality drops massively without it
