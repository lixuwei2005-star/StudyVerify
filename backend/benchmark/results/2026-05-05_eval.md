### Hypothesis

1. **`entry_function` naming drift**: Solver-generated reference may use camelCase (`twoSum`) while dataset uses snake_case (`two_sum`), causing the sandbox to fail to find the function in the student's code path.
2. **Verifier LLM over-rejects**: The Step 6.4 anti-leak retry layer may over-trigger on "looks-similar" code, judging correct solutions as failed.

Both are testable in Step 10 with a small repro suite (re-verify each of the 60 reference-fails with naming variations and prompt ablations). Given the verifier handles variants perfectly, the fault is on the verifier's pass-detection path, not its bug-detection path.

## Anti-leak success

**70.6%** combined (phrase filter AND LLM judge both pass). The two layers serve different roles:

- **Phrase filter (93.8%)** — 33-phrase substring scan. Catches literal forbidden phrases (`loop through`, `use sum()`, `running total`, `recursion`, etc.). Necessary but insufficient — most leakage is structural, not lexical.
- **LLM judge (75.2%)** — DeepSeek-as-judge, prompted to assess whether a hint reveals algorithm syntax or named Python constructs. Catches subtle leakage the phrase filter misses.

### Phrase filter passed; LLM judge caught the leak (real samples from this run)

- **`lc-001-two-sum` :: variant `subtraction-instead-of-addition` :: hint #2**
  > Your condition checks if `n - target` is in the dictionary. Is that the number that would pair with n to make the target? What number plus n equals the target?
  - **judge reason:** hint reveals use of dictionary and complement check

- **`lc-001-two-sum` :: variant `subtraction-instead-of-addition` :: hint #4**
  > Your code currently looks for `n - target`, but the complement that adds with `n` to reach `target` is found by `target - n`. Which subtraction gives you the correct pair number?
  - **judge reason:** Reveals exact subtraction formula for complement.

- **`lc-344-reverse-string` :: variant `returns-input-unchanged` :: hint #1**
  > Python strings support a slice syntax with a third argument called the step. What step value would produce the characters in reverse order?
  - **judge reason:** Reveals slice syntax and step parameter.

These each pass the static phrase list (no `loop`, `iterate`, `use X`) yet reveal algorithm structure (dictionary lookup, slice notation, loop iteration) when read in context. The LLM judge captures that context the phrase filter cannot.

### Interpretation

Algorithmic-pattern problems (`recursion`, `two-pointers`) push hints toward syntax — the algorithm itself is close to the implementation, leaving the hint generator little room to be Socratic without naming the technique. Data-shape problems (`array`, `string`, `math`) admit more conceptual hints.

**Improvement direction** (Step 10): topic-specific prompt iteration. For algorithmic patterns, add an explicit constraint to the hint prompt: "Don't reference algorithmic techniques (recursion, pointers, binary search) by name; describe the *invariant* the student should track instead."

## Helpfulness progression

Quote-gated metric: an LLM-student simulator reads the buggy code and the hint, then proposes a change. The hint counts as "helpful" only if the simulator's `proposed_change` contains ≥5 consecutive word-tokens from the hint text. This forces the metric from "LLM says yes" to "LLM demonstrably used the hint" — Phase A's smoke saturated at 100% across all hint levels because the simulator was drawing on its own training data, not the hint.

| Hint level | Would-fix rate (quote-gated) |
|---|---|
| 1 (most Socratic) | 91.9% |
| 2 | 98.2% |
| 3 | 97.9% |
| 4 | 98.6% |
| 5 (most directive) | 96.1% |

Progression visible: hint 1 has the lowest would-fix rate (91.9%) — Socratic phrasing makes it harder for the student-sim to quote five consecutive words. Hints 2–5 hover around 98% — more directive, easier to quote.

**Saturation observation**: 96–98% across hints 2–5 suggests either (a) hints converge to algorithmic guidance quickly, or (b) the 5-word quote threshold is too generous. The first interpretation is consistent with the anti-leak finding above (algorithmic patterns leak more) — they're the same underlying signal viewed from two angles. **Step 10 future work**: tighten quote-gate to 10+ words for sharper progression.

## What this evaluation surfaced

Production-mode benchmark catches signals that unit tests can't:

1. **Asymmetric verifier (Finding 1)**: 60 of 95 valid reference solutions judged FAIL (63% false-reject rate), while every variant bug (285/285) was caught. The verifier knows what's wrong; it isn't sure what's right. Step 10 closure target.
2. **Anti-leak gap by topic (Finding 3)**: recursion (44% leak rate) and two-pointers (36% leak rate) leak the most; algorithmic-pattern problems push hints toward syntax. Topic-specific prompt iteration scheduled.
3. **Helpfulness measurement requires the quote-gate (Finding 2)**: Phase A's smoke saturated at 100% across all hint levels because the LLM-student leaned on training data instead of the hint. Adding a 5-word literal-quote requirement recovered a real progression signal — hint 1 dips, hints 2–5 plateau.
4. **Reliability is workable (Finding 4)**: 99.6% end-to-end across 2,000+ calls, with no `/verify` failures and only 2 `/hint` failures. Solver is the soft spot.
